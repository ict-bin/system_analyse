# 你的工作分两个阶段

## Phase A: 文件分类

你会收到一个解包后的软件包目录，需要：
1. 遍历所有文件，分析每个文件的类型和功能
2. 按功能将文件划分为逻辑模块（如 network、crypto、init、web_server 等）
3. 为每个模块创建子目录，在其中创建 `files.list`，每行记录一个文件的相对路径（相对于 target/）
4. **不要拷贝文件，只记录路径到 files.list**
5. 确保每个文件都被分类，不遗漏

### 分类原则
- 按功能聚合，而非按文件类型
- 配置文件归入其配置的功能模块
- 共享库/工具文件可归入 `common` 或 `utils` 模块
- 无法确定功能的文件归入 `unknown` 模块

## Phase B: 逐模块分析

对每个模块独立完成分析，将结果写入该模块目录的 `module_report.md`：

分析时先读取 `files.list` 获取文件列表，再从 `target/` 目录读取每个源文件。

### 1. 模块功能分析
- 模块包含的文件及各文件作用
- 模块整体功能和职责
- 模块对外提供的接口/服务

### 2. 威胁分析 (STRIDE)
| 类别 | 检查项 |
|------|--------|
| **S** Spoofing | 身份伪造、认证绕过 |
| **T** Tampering | 缓冲区溢出、未校验的输入 |
| **R** Repudiation | 缺乏日志/审计 |
| **I** Info Disclosure | 敏感数据泄露、错误信息过详 |
| **D** DoS | 资源耗尽、崩溃、无限循环 |
| **E** EoP | 命令注入、权限升级 |

每个威胁必须标注：
- 位置（文件:行号或配置项）
- 触发条件
- 影响
- 风险等级：🔴 高 / 🟡 中 / 🟢 低

### 3. 对外暴露面评估
- 网络端口、文件路径、IPC 通道
- 综合风险评分 (0-100)

# 质量要求

1. **引用具体位置**：每个发现必须标注文件名和行号/配置项
2. **不臆造**：只报告确实存在的问题
3. **不遗漏**：使用 `read` 工具逐个读取文件后再分析
4. **可操作**：修复建议必须具体
