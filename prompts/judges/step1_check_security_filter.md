# 安全维度过滤评审 Judge

## 评审目标

验证 Worker 的安全维度过滤操作是否准确：**没有误删相关模块，没有漏删无关模块。**

## 评审步骤

### Step 1：了解过滤前后差异

比对两个目录（代码中已在 prompt 中列出）：
- **备份目录**（过滤前）：`modules_pre_filter_backup/`
- **当前目录**（过滤后）：`modules/`

如需查看某模块的文件列表：
```bash
# 被删除的模块（在备份中查看）
cat modules_pre_filter_backup/<模块名>/files.list

# 被保留的模块
cat modules/<模块名>/files.list
```

### Step 2：验证被删除的模块

对每个**被删除的模块**：
- 读取备份中的 `files.list`
- 判断该模块是否确实与指定安全维度无关
- 若发现**误删**（模块应保留）→ 在意见中明确指出

### Step 3：验证被保留的模块

对**被保留的模块**抽查（重点检查与维度关联性最弱的）：
- 是否与指定安全维度有实质关联
- 若发现**漏删**（模块应删除）→ 在意见中明确指出

### Step 4：一致性检查

- `modules/` 目录不能为空（至少保留1个模块）
- 不应存在既在 `modules/` 又在 `modules_pre_filter_backup/` 且内容不同的情况（Worker 不应修改文件内容，只删除目录）

## 评分标准

| 情况 | 分数 |
|------|------|
| 过滤准确，无误删、无漏删 | 90-100 |
| 1-2 个轻微误判（如保留了关联度较低但无害的模块） | 70-89 |
| 有明显误删（删除了应保留的相关模块） | 0-49 |
| `modules/` 为空 | 0 |

**通过条件：评分 ≥ 70 且 `modules/` 目录非空。**

## 输出格式

```markdown
## 评分: <0-100>

## 通过: <是/否>

## 评审意见:
<逐条说明保留/删除决策的合理性，列出误删或漏删的模块（如有）>
```
